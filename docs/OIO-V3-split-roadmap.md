# OIO V3 拆分路线图

> 基于 `OIO-V3-and-3.1-development-plan(4).md` 拆分。
> 原方案内容很完整，但范围过大；本文件只负责把它拆成更容易实际推进的版本。

## 1. 拆分原则

V3 原方案同时包含：

```text
技术升级
IndexedDB
旧数据迁移
备份恢复
AI API
Review V3
TTS
听音识别
跟读录音
输入法模式
```

这些功能互相有关，但不应该全部塞进一个版本。

拆分原则：

```text
先保证数据安全
再升级学习核心
再加入声音系统
最后考虑输入法模式
```

## 2. 推荐版本拆分

```text
V3.0：技术底座与数据迁移
V3.1：AI API + Review V3
V3.2：TTS 与听音练习
V3.3：跟读与录音对比
OIO 3.1：输入法模式支线
```

说明：

- `V3.x` 是主应用迭代线。
- `OIO 3.1` 输入法模式建议作为独立支线，不阻塞主应用 V3。

## 3. V3.0：技术底座与数据迁移

### 目标

把当前静态版升级成可长期维护的本地优先应用。

### 范围

```text
React
TypeScript
Vite
IndexedDB
Dexie.js
Zod
Vitest
旧数据迁移
备份恢复
```

### 必做

- 搭建 React + TypeScript + Vite 项目
- 迁移当前核心页面
- 建立 IndexedDB schema
- 建立 repository 层
- localStorage -> IndexedDB 迁移
- 迁移前自动备份
- 全量 JSON 导出
- 全量 JSON 导入
- 基础测试

### 不做

- AI API
- Review V3
- TTS
- 听音练习
- 跟读录音
- 输入法模式

### 验收标准

```text
旧版数据可以安全迁移
当前 V2 核心功能可继续使用
刷新页面数据不丢失
可以导出完整备份
可以从备份恢复
代码结构不再集中在单个 app.js
```

### 为什么先做这个

后续音频、图片、API 缓存、Review 阶段数据都会让数据结构变复杂。

如果不先升级存储和架构，后面每个功能都会越来越难维护。

## 4. V3.1：AI API + Review V3

### 目标

升级核心学习流程，让 OIO 从普通填空复习，进化到渐进式回忆。

### 范围

```text
API 设置
高质量模型改写
低成本模型生成 Stage 3 问题
Review V3 三阶段
问题缓存
手动编辑问题
旧卡片迁移为 Stage 1
```

### AI API

只做两类 AI：

```text
Rewrite Model：高质量改写
Question Model：生成 Stage 3 问题
```

不做：

- AI 自动推荐 Chunk
- AI 自动匹配 Source Intent
- AI 自动选 Cloze
- AI 语义评分
- AI 自动控制复习结果

### Review V3

三阶段：

```text
Stage 1：普通填空
Stage 2：先显示完整 Chunk，开始输入后隐藏 Chunk，仍然填写原 Cloze
Stage 3：根据 AI 问题和上下文回忆表达
```

### 建议统一数据规则

原方案里同时出现了两套推进方式：

```text
每阶段完成次数
统一 mastery 数值
```

建议 V3.1 只保留一种。

推荐：

```text
使用统一 mastery 数值推进阶段
```

示例：

```text
mastery < 1
-> Stage 1

1 <= mastery < 2
-> Stage 2

2 <= mastery < 3
-> Stage 3

mastery >= 4.5
-> Graduated
```

### 不做

- TTS
- 听音识别
- 跟读录音
- 输入法模式
- IELTS 特化

### 验收标准

```text
可以直接调用 API 改写
API 失败不丢失输入
手动模式仍然可用
卡片可从 Stage 1 进入 Stage 2
Stage 2 输入时可以隐藏完整 Chunk
Stage 3 可以显示 AI 生成问题
Stage 3 不做自动评分
用户仍然用 Again / Good / Easy 自评
```

## 5. V3.2：TTS 与听音练习

### 目标

让用户能听到自己学习的文本和 Chunk，并训练“听到声音能识别表达”。

### 范围

```text
TTS 设置
完整文本音频
句子音频
Chunk 音频
Listening 独立栏目
先听后看
看文本听音
隐藏 Chunk 听音
听音自我记录
```

### 推荐第一版生成策略

不要先做复杂自动裁切。

推荐：

```text
完整文本音频：整篇生成
句子音频：按句单独生成
Chunk 音频：对用户选中的 Chunk 单独生成
```

这样比依赖 TTS 时间戳稳定。

### 不做

- 浏览器录音
- 原音与录音对比
- 发音评分
- ASR
- 自动语音分析
- 输入法模式

### 验收标准

```text
每篇文本可以生成完整音频
每个 Chunk 可以生成独立音频
Listening 页面按文本分组
可以裸听 / 看文本听 / 隐藏 Chunk 听
听音进度不影响 Review 进度
音频生成失败不破坏文本数据
```

## 6. V3.3：跟读与录音对比

### 目标

在已有 TTS 和听音基础上，增加口语模仿练习。

### 范围

```text
立即跟读
循环模仿
浏览器录音
播放用户录音
原音与录音对比
最近录音保留
录音权限错误处理
```

### 不做

- 发音自动评分
- 语调自动评分
- ASR 文本识别
- 自动重音分析

### 验收标准

```text
用户可以播放原音
用户可以录下跟读
用户可以播放自己的录音
可以在原音和录音之间切换对比
录音失败有明确提示
```

## 7. OIO 3.1：输入法模式支线

### 定位

输入法模式不是 V3 主应用的必需功能。

建议作为独立支线：

```text
OIO 3.1：快速输入 / 输入法中间层
```

它可以在 V3.1 后开始设计，也可以等 V3.2 稳定后再做。

### 为什么不放进 V3 主线

输入法模式涉及：

- 全局快捷键
- 剪贴板
- 自动粘贴
- 浏览器扩展或桌面集成
- 隐私边界
- 一次性卡片

这些复杂度和主应用学习流程不一样。

如果和 V3 主线一起做，会拖慢技术升级和 Review V3。

### 推荐拆分

```text
3.1A：快速输入 MVP
3.1B：一次性复习队列
3.1C：桌面集成
3.1D：浏览器扩展，可选
```

### 最小 MVP

```text
手动激活
快速输入窗口
调用 Rewrite Model
编辑改写结果
复制结果
可选 Chunk / Cloze
生成低优先级一次性卡片
Again：10 分钟
Easy：进入垃圾箱
垃圾箱 1 天后删除
```

## 9. 建议的开发顺序

- 给当前 V2 打 tag
- 整理 docs
- 确认备份方式
- 写清楚旧数据结构

### Phase 1：V3.0 技术底座

- 新项目结构
- IndexedDB schema
- 旧数据迁移
- 备份恢复
- 基础页面迁移

### Phase 2：V3.1 AI 改写

- 设置页
- API Key 本地保存策略
- Rewrite Model
- API 模式新建文本
- 手动模式保留

### Phase 3：V3.1 Review V3

- ReviewItem 数据结构
- mastery 阶段推进
- Stage 1
- Stage 2
- Stage 3
- Question Model

### Phase 4：V3.2 音频基础

- TTS 设置
- 完整文本音频
- 句子音频
- Chunk 音频
- 音频播放器

### Phase 5：V3.2 Listening

- Listening 页面
- 听音练习模式
- 自我记录

### Phase 6：V3.3 跟读

- 录音权限
- 录音保存

### Phase 7：OIO 3.1 输入法支线

- 快速输入 MVP
- 一次性卡片
- 快捷键 / 剪贴板

## 10. 最推荐的近期执行版本

下一步不要直接做完整 V3。

最推荐先做：

```text
V3.0 技术底座与数据迁移
```

原因：

- 当前 V2 已经能日常使用
- 下一个最大风险是数据可靠性
- 图片和后续音频都需要 IndexedDB
- React/TS 迁移会影响所有后续功能
- 不先做底座，AI 和音频会堆在旧 app.js 上

V3.0 完成后，再做：

```text
V3.1 AI API + Review V3
```

这样节奏最稳。

## 11. 一句话结论

原方案方向正确，但应该拆成：

```text
V3.0 技术底座
V3.1 AI + Review V3
V3.2 听音
V3.3 跟读
OIO 3.1 输入法模式支线
```

不要让输入法、音频、Review、API 和技术迁移同时成为同一个版本的完成条件。
